# 秒杀系统技术方案

## 一、业务背景

### 1.1 业务场景
秒杀系统是一种高并发、高流量的电商促销场景，具有以下特点：
- **高并发**：短时间内大量用户同时抢购
- **库存有限**：商品数量有限，售完即止
- **限购规则**：每个用户仅限购买一件商品
- **时效性强**：下单后需在规定时间内完成支付，超时自动关闭订单

### 1.2 核心需求
- **高性能**：支持万级QPS的秒杀请求
- **数据一致性**：保证Redis缓存库存与MySQL真实库存的一致性
- **幂等性**：防止用户重复下单
- **可靠性**：异常情况下能够补偿处理
- **轻量化**：避免使用分布式锁，保持架构简洁

## 三、核心流程

### 3.1 秒杀下单流程

```
用户发起秒杀请求
    │
    ▼
┌─────────────────────────────────────────┐
│ 1. Redis Lua脚本原子化检查              │
│    - 检查用户是否已限购                  │
│    - 检查Redis缓存库存是否充足            │
│    - 预扣Redis缓存库存                   │
│    - 添加用户限购标记                    │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│ 2. 发送Kafka消息                         │
│    消息体: {userId, goodsId}             │
│    不包含orderId (消费端生成)            │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│ 3. 立即返回响应                          │
│    "下单已受理，请15分钟内完成支付"       │
└─────────────────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│ 4. Kafka消费端处理                       │
│    - 检查幂等性 (user_id, goods_id)      │
│    - 本地生成全局唯一orderId             │
│    - 开启本地事务                        │
│    - 创建订单 (status=UN_PAY)           │
│    - 扣减MySQL真实库存                   │
│    - 提交事务                            │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│ 5. 异常处理                              │
│    - 本地重试2次 (间隔1s/3s)             │
│    - 失败后写入异常日志表                │
│    - 触发告警                            │
└─────────────────────────────────────────┘
```

### 3.2 支付回调流程

```
支付平台发起回调
    │
    ▼
┌─────────────────────────────────────────┐
│ 1. 查询订单                              │
│    - 重试3次 (间隔1s/2s/4s)              │
│    - 查询条件: order_id                  │
└──────────────┬──────────────────────────┘
               │
               ▼
        ┌──────┴──────┐
        │             │
    查到订单      未查到订单
        │             │
        ▼             ▼
┌──────────────┐  ┌──────────────┐
│ 2. 乐观锁更新 │  │ 直接原路退款 │
│    订单状态   │  └──────────────┘
│    UN_PAY→PAYED│
│    记录支付时间 │
└──────────────┘
```

### 3.3 超时关单流程

```
XXL-Job定时任务触发 (每分钟)
    │
    ▼
┌─────────────────────────────────────────┐
│ 1. 查询超时未支付订单                    │
│    条件: status=UN_PAY                  │
│          AND pay_deadline < NOW()       │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│ 2. 开启事务处理                          │
│    - 乐观锁关闭订单                      │
│      (status: UN_PAY→CANCELED)          │
│    - 恢复MySQL真实库存                   │
│    - 恢复Redis缓存库存                   │
│    - 移除用户限购标记                    │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│ 3. 提交事务                              │
│    保证数据一致性                        │
└─────────────────────────────────────────┘
```

### 3.4 异常补偿流程

```
XXL-Job定时任务触发 (每分钟)
    │
    ▼
┌─────────────────────────────────────────┐
│ 1. 查询未处理的异常日志                  │
│    条件: handle_status=0                │
│          AND retry_count < 5            │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│ 2. 重新执行建单逻辑                      │
│    - 生成orderId                         │
│    - 创建订单                            │
│    - 扣减库存                            │
└──────────────┬──────────────────────────┘
               │
               ▼
        ┌──────┴──────┐
        │             │
     成功处理       处理失败
        │             │
        ▼             ▼
┌──────────────┐  ┌──────────────┐
│ 标记已处理    │  │ 增加重试次数 │
│ handle_status│  │ 达到最大次数 │
│ = 1          │  │ 标记人工处理 │
└──────────────┘  │ handle_status│
                  │ = 2          │
                  └──────────────┘
```

## 四、对外HTTP接口

**请求参数**:
```json
{
  "userId": 123456,
```

```json
{
  "code": 200,
  "message": "下单已受理，请15分钟内完成支付",

  "data": null
```

```json
  "message": "每人仅限秒杀一单",
  "data": null
```

### 5.2 Kafka配置 (KafkaConfig.java)

```java
@EnableKafka
@Configuration
public class KafkaConfig {
    @Value("${spring.kafka.bootstrap-servers}")
    private String bootstrapServers;

    // 生产者配置
    @Bean
    public ProducerFactory<String, KafkaMsgDTO> producerFactory() {
        Map<String, Object> config = new HashMap<>();
        config.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, bootstrapServers);
        config.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        config.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, JsonSerializer.class);
        return new DefaultKafkaProducerFactory<>(config);
    }

    // 消费者配置
    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, KafkaMsgDTO> kafkaListenerContainerFactory() {
        ConcurrentKafkaListenerContainerFactory<String, KafkaMsgDTO> factory = new ConcurrentKafkaListenerContainerFactory<>();
        Map<String, Object> config = new HashMap<>();
        config.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, bootstrapServers);
        config.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        config.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, JsonDeserializer.class);
        config.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, "latest");
        config.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, "false"); // 手动提交offset
        factory.setConsumerFactory(new DefaultKafkaConsumerFactory<>(config));
        return factory;
    }
}
```

### 5.4 Redis配置 (RedisConfig.java)

```java
@Configuration
public class RedisConfig {
    @Bean
    public RedisTemplate<String, String> redisTemplate(RedisConnectionFactory factory) {
        RedisTemplate<String, String> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        template.setKeySerializer(new StringRedisSerializer());
        template.setValueSerializer(new StringRedisSerializer());
        template.setHashKeySerializer(new StringRedisSerializer());
        template.setHashValueSerializer(new StringRedisSerializer());
        template.afterPropertiesSet();
        return template;
    }
}
```

## 六、数据库设计

### 6.1 商品表 (t_goods)

```sql
CREATE TABLE t_goods (
    id BIGINT PRIMARY KEY COMMENT '商品ID',
    stock INT NOT NULL DEFAULT 0 COMMENT '真实库存',
    name VARCHAR(100) NOT NULL COMMENT '商品名称',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='秒杀商品表';
```

### 6.2 订单表 (t_order)

```sql
CREATE TABLE t_order (
    order_id VARCHAR(64) PRIMARY KEY COMMENT '订单号（消费端生成）',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    goods_id BIGINT NOT NULL COMMENT '商品ID',
    status VARCHAR(20) NOT NULL DEFAULT 'UN_PAY' COMMENT 'UN_PAY未支付 / PAYED已支付 / CANCELED已取消',
    pay_deadline DATETIME NOT NULL COMMENT '15分钟支付截止时间',
    pay_time DATETIME NULL COMMENT '支付时间',
    close_time DATETIME NULL COMMENT '关单时间',
    out_pay_no VARCHAR(64) NULL COMMENT '外部支付流水号',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_goods (user_id, goods_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='秒杀订单表';
```

### 6.3 Kafka消费异常日志表 (t_seckill_mq_fail_log)

```sql
CREATE TABLE t_seckill_mq_fail_log (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    goods_id BIGINT NOT NULL,
    msg_content TEXT COMMENT '原始消息',
    error_msg TEXT COMMENT '异常信息',
    retry_count INT NOT NULL DEFAULT 0 COMMENT '已重试次数',
    handle_status TINYINT NOT NULL DEFAULT 0 COMMENT '0未处理 1已处理 2人工处理',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_handle_status (handle_status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='Kafka消费异常日志';
```

## 十、总结

本秒杀系统采用极简架构设计，通过Redis+Lua脚本实现高性能库存预扣，Kafka异步解耦提升系统吞吐量，数据库唯一索引保证幂等性，XXL-Job定时任务实现异常补偿。整体架构轻量化、高可靠、易扩展，能够满足高并发秒杀场景的业务需求。
